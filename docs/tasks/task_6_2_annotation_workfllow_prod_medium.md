# task_6_2_medium: Production Annotation Handoff

> This task belongs to the pickup/putdown temporal action detection implementation.
> Read `docs/CONCEPTS.md`, `docs/LABELING_GUIDELINES.md`, `docs/ANNOTATION_WORKFLOW.md`, and `docs/IMPLEMENTATION_PLAN.md` before starting.

**Task ID:** `task_6_2`
**Difficulty:** `medium`
**Dependencies:** Tasks 5, 6, and 6.1
**Parallel work:** Task 7 may begin after the metadata and export contracts are stable.

## Objective

Bridge remotely generated candidate clips into the existing Label Studio workflow.

The implementation must create Label Studio tasks from uploaded candidate metadata and export annotations using timestamps relative to the original source video rather than the candidate clip.

## Inputs

* Uploaded candidate videos
* Candidate metadata generated by Task 6.1
* Existing Label Studio configuration from Task 6
* Existing annotation task builder and exporter
* Candidate-to-source timing information

## Required candidate metadata

Each imported Label Studio task must preserve, where available:

* `candidate_id`
* `clip_id` — identifier of the original source video
* `source_start_s`
* `source_end_s`
* candidate video S3 key or Label Studio-accessible URL
* `actor_id`
* `hand_side`
* `region_id`
* proposal or candidate score
* generation configuration fingerprint

Missing optional metadata must not prevent task creation, but required timing and identity fields must produce a clear validation error.

## Functional requirements

### 1. Build Label Studio tasks

Extend the existing annotation task builder or add a focused adapter that reads Task 6.1 candidate metadata.

Each generated task must contain:

```json
{
  "data": {
    "video": "<Label Studio accessible candidate URL>",
    "candidate_id": "candidate_001",
    "clip_id": "source_clip_001",
    "source_start_s": 102.0,
    "source_end_s": 110.0,
    "actor_id": "track_3",
    "hand_side": "right",
    "region_id": "shelf_2"
  }
}
```

The exact JSON layout may follow existing repository conventions, but all source-mapping fields must survive the Label Studio import/export round trip.

### 2. Do not assign a default event label

Candidate generation identifies a possible interaction interval, not an event class.

The task builder must not automatically label candidates as `pickup`, `putdown`, or `ignore`.

Annotators must explicitly choose the event type.

### 3. Convert timestamps during export

Label Studio annotations are relative to the candidate video.

The exporter must convert them to original source-video timestamps:

```text
source_event_start_s =
    source_start_s + candidate_relative_start_s

source_event_end_s =
    source_start_s + candidate_relative_end_s
```

Example:

```text
Candidate source interval: 102.0–110.0 seconds
Annotation inside candidate: 2.3–3.7 seconds
Canonical source event: 104.3–105.7 seconds
```

The canonical exported event must retain both:

* the original source `clip_id`;
* the originating `candidate_id`, for traceability.

### 4. Validate timestamp bounds

Reject or report annotations where:

* `source_start_s >= source_end_s`;
* relative event start is negative;
* relative event end exceeds candidate duration beyond the allowed tolerance;
* event start is not before event end;
* required source mapping fields are missing.

Small floating-point or frame-boundary differences may use an explicitly documented tolerance.

### 5. Preserve existing annotation behavior

The implementation must continue to support:

* `pickup`
* `putdown`
* `ignore`
* confidence
* hard-case status
* item count
* review status
* notes
* deterministic multi-item row expansion
* ignore interval export

Existing local annotation workflows must remain functional.

## Expected outputs

The task should produce:

```text
annotation/tasks_candidates.json
annotation/exports/events.csv
annotation/exports/ignore_intervals.parquet
```

Exact output locations may follow existing configuration conventions.

The generated `events.csv` must use source-video timestamps.

## CLI expectations

Prefer extending the existing commands rather than creating duplicate annotation pipelines.

Example interface:

```bash
pickup-putdown annotation-build-tasks \
  --candidate-metadata-dir .local/candidate_staging/metadata \
  --output annotation/tasks_candidates.json
```

```bash
pickup-putdown annotation-export \
  --input annotation/label_studio_export.json \
  --events-output annotation/exports/events.csv \
  --ignore-output annotation/exports/ignore_intervals.parquet
```

The implementation may use different option names when required by the existing CLI structure.

## Tests

Add automated tests covering:

1. Task JSON contains all required source metadata.
2. No default `pickup` or `putdown` label is created.
3. A zero-offset candidate preserves annotation timestamps.
4. A non-zero offset is added correctly.
5. Events at candidate boundaries are converted correctly.
6. Invalid relative timestamps are rejected.
7. Missing required metadata produces a clear error.
8. Pickup and putdown annotations export correctly.
9. Ignore intervals receive the same source offset conversion.
10. Multi-item annotations remain deterministic.
11. Actor, hand, and region metadata survive the round trip.
12. Repeated exports produce identical canonical output.

Include at least one end-to-end fixture:

```text
candidate metadata
→ Label Studio task JSON
→ simulated Label Studio annotation
→ canonical events.csv
```

## Pilot annotation

After automated tests pass:

1. Generate a batch of approximately 30–50 representative candidate tasks.
2. Import them into the production-like Label Studio setup.
3. Annotate pickups, putdowns, negatives, and hard cases.
4. Export the annotations.
5. Validate the canonical files.
6. Manually verify several exported timestamps against the original source videos.
7. Record any workflow or schema problems before bulk annotation begins.

The pilot should include:

* candidates with non-zero source offsets;
* candidates close to source-video boundaries;
* clear pickups and putdowns;
* candidates containing no valid event;
* multiple-person or ambiguous cases;
* at least one ignore interval;
* multi-item events where available.

## Out of scope

This task does not include:

* full active-span annotation;
* proposal-recall measurement;
* train/validation/test splitting;
* dataset version freezing;
* model training;
* candidate-generation algorithm changes;
* automated event classification.

These belong to Task 7 or later model tasks.

## Definition of done

The task is complete when:

* uploaded candidate metadata can be converted into valid Label Studio tasks;
* no event class is assigned automatically;
* annotations can be exported successfully;
* exported event timestamps correspond to the original source video;
* candidate and source identifiers remain traceable;
* required round-trip tests pass;
* a 30–50 candidate pilot has been completed;
* manually checked events align with the correct moments in their original source videos;
* the production annotation workflow is documented.
